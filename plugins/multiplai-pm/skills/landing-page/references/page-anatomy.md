# Landing Page Anatomy Reference

The universal 8-section structure for high-converting landing pages. Load this file when creating or auditing landing page copy.

---

## The 8-Section Structure

This structure mirrors the buyer's decision process and is consistent across every credible CRO source. The framework (AIDA, PAS, Pierri) changes the copy approach within sections, but the section order stays the same.

```
1. NAVBAR         → "Where am I?"
2. HERO           → "What is this and why should I care?"
3. SOCIAL PROOF 1 → "Can I trust this?"
4. BENEFITS       → "How does it help me?"
5. SOCIAL PROOF 2 → "Do others like me use this?"
6. FAQ            → "What about [objection]?"
7. FINAL CTA      → "What do I do next?"
8. FOOTER         → "Is this legitimate?"
```

---

## Section-by-Section Guide

### 1. Navbar

**Purpose:** Minimal orientation. NOT navigation.

**Rules:**
- Logo (links to homepage) on the left
- One CTA button on the right (same as the page's primary CTA)
- No dropdown menus, no full site navigation
- Every nav link is an exit — remove them on dedicated landing pages

**Copy needed:** Just the CTA button text.

---

### 2. Hero

**Purpose:** Answer "What is this and why should I care?" in 5 seconds.

**Elements:**
- **Headline** (H1) — the single most important line of copy on the page
- **Subheadline** — expands on the headline with specifics or qualification
- **Primary CTA button** — the desired action
- **CTA microcopy** — reduces friction ("No credit card required")
- **Visual** — product screenshot, demo, hero image, or illustration

**Framework-Specific Hero Approaches:**

| Framework | Headline Angle | Subheadline Angle |
|-----------|---------------|-------------------|
| **AIDA** | Attention-grabbing outcome | Build interest with specifics |
| **PAS** | Name the problem directly | Agitate the consequences |
| **Pierri** (B2B SaaS) | [Audience] + [Category] + [Outcome] | Context → Problem → Capability |
| **BAB** | Paint the "after" state | Bridge from current pain |

**Headline Formulas:**
- "{Achieve outcome} without {pain point}"
- "The {category} for {audience}"
- "Never {unpleasant event} again"
- "{Specific number} {audience} use {product} to {outcome}"
- "{Question highlighting main pain point}?"

**Do:**
- Front-load the value proposition — don't make them scroll to understand
- Use the customer's language, not your internal terminology
- Include a visual that shows the product in action (screenshots > stock photos)

**Don't:**
- Lead with the company name or mission statement
- Use abstract buzzwords ("innovative solutions for modern teams")
- Put a form here — just the CTA button (forms go lower)

---

### 3. Social Proof #1

**Purpose:** Establish immediate credibility after the claim.

**Elements (choose based on maturity stage):**

| Stage | What to show |
|-------|-------------|
| **Pre-launch** | Founder credibility ("Built by a 20-year veteran of..."), technical specifics, product architecture |
| **Early** | Beta user count, waitlist size ("Join 340 early adopters") |
| **Growing** | Customer logos, user count, press mentions |
| **Established** | Logo bar ("Trusted by"), "As Seen In" press logos, specific stats |

**Placement:** Directly below the hero. This is the "trust bridge" — the hero made a promise, social proof validates it.

**Copy needed:** Minimal. A label line ("Trusted by teams at") plus logos, or a specific stat with context.

---

### 4. Benefits & Objection Handling

**Purpose:** Show how the product solves their problem with specific outcomes.

**Structure:** 2-4 benefit blocks, each following:
- **Benefit headline** — outcome-focused, not feature-focused
- **Benefit description** — 2-3 sentences explaining the "how" behind the outcome
- **Supporting visual** — screenshot, icon, or diagram (optional but helps)

**Transform features into benefits:**
- Feature: "AI-powered analytics" → Benefit: "Know which customers will churn before they leave"
- Feature: "Real-time collaboration" → Benefit: "Your whole team works on the same page, literally"

**Framework-Specific Approaches:**

| Framework | How to structure benefits |
|-----------|-------------------------|
| **AIDA** | Build interest → create desire. Each benefit deepens engagement. |
| **PAS** | Present the solution. Each benefit directly addresses a pain from the hero. |
| **Pierri** | Capability → Feature → Benefit chain. Show how the product works, then why it matters. |

**Objection handling within benefits:**
Weave in common objections naturally. If the objection is "Is this hard to set up?", one benefit should be "Up and running in 15 minutes" with a description that explains the setup process.

**Do:**
- Lead with the outcome, not the mechanism
- Use specific numbers when available ("4 hours → 15 minutes")
- Address one benefit per block — don't bundle

**Don't:**
- List features without tying them to outcomes
- Use more than 4 benefit blocks (pick the strongest)
- Use abstract icons that don't relate to the actual benefit

---

### 5. Social Proof #2

**Purpose:** Deepen trust with evidence from real users.

**Elements (choose based on maturity stage):**

| Stage | What to show |
|-------|-------------|
| **Pre-launch** | Product demo/video, technical proof (architecture, benchmarks), founder's personal story |
| **Early** | 1-2 testimonials from beta users, specific results achieved, before/after comparison |
| **Growing** | 3-4 testimonials, case study summary, specific metrics |
| **Established** | Rich case studies, video testimonials, detailed metrics, expert endorsements |

**Testimonial Copy Formula:**
"[Specific problem they had]. [What they tried before]. [How this product solved it]. [Specific result]."

**Example:**
"We were spending 6 hours a week on invoice follow-ups. Tried three different tools, all too complex. With Acme, we send invoices in 30 seconds and get paid in half the time. Last quarter our AR dropped from 45 days to 18."

**Credibility boosters:**
- Full name + title + company (or LinkedIn link for B2B)
- Photo (real, not stock)
- Specific metrics in the quote
- Minor critiques included ("It took a week to get used to the interface, but after that...")

**Do:**
- Use quotes that include specific details and results
- Match testimonial persona to target visitor persona
- Place near the benefits they validate

**Don't:**
- Fabricate testimonials (ever — even as placeholders)
- Use only first names without context ("— Sarah K.")
- Cherry-pick only perfect reviews (minor critiques increase credibility)

---

### 6. FAQ

**Purpose:** Handle remaining objections. This section gets 17% of all page clicks — visitors actively seek it.

**Structure:** 5-8 questions in expandable accordion format.

**What to include:**
- The objections you'd handle in a sales call
- Pricing/commitment questions ("Is there a free tier?" "Can I cancel anytime?")
- Technical requirements ("Do I need to install anything?")
- Comparison questions ("How is this different from X?")
- Security/privacy concerns (if relevant)

**Copy approach:**
- Questions in the customer's voice ("What if I need to cancel?")
- Answers that are direct and specific, not defensive
- Keep answers to 2-3 sentences each

**Do:**
- Lead with the most common objection
- Include a CTA in or near the FAQ ("Still have questions? Talk to us")

**Don't:**
- Use FAQ as a dumping ground for miscellaneous info
- Write overly long answers (that's what docs are for)
- Skip this section — it's one of the most clicked areas on any landing page

---

### 7. Final CTA

**Purpose:** Convert visitors who've read the whole page. These are the most convinced visitors — give them a clear action.

**Elements:**
- **Headline** — restate the core value proposition in a new way, or use urgency
- **CTA button** — same action as the hero, but copy can vary
- **Microcopy** — reinforce trust ("30-day money-back guarantee", "No credit card needed")
- **Optional: email capture form** — if the CTA involves sign-up

**Copy approach:**
- This is where you can use mild urgency (deadlines, limited spots) — but only if real
- Loss aversion framing works well here: "Don't let another week of [pain] go by"
- Restate one key social proof stat near the button

**Do:**
- Make the CTA button impossible to miss (visual contrast)
- Use different phrasing from the hero CTA to address different motivations
- Keep the form short (name + email maximum for most products)

**Don't:**
- Introduce new information — this section is about action, not education
- Use fake urgency or manufactured scarcity
- Add long forms with many fields (120% conversion lift from reducing fields)

---

### 8. Footer

**Purpose:** Legitimacy signals and legal compliance.

**Elements:**
- Company name and brief description
- Links: Privacy Policy, Terms of Service, Contact
- Social media links (optional)
- Physical address (increases trust, required in some jurisdictions)

**Copy needed:** Minimal. A one-line company description and standard links.

## Stage-Adaptive Anatomy

The 8-section structure adapts based on product maturity, detected during discovery:

### Pre-Launch (No Users)

No customer social proof exists. Substitute with:

| Section | Adaptation |
|---------|-----------|
| Social Proof #1 | Founder credibility: "Built by [name], [X]-year [domain] veteran" + product specifics |
| Social Proof #2 | Live demo / product screenshots / technical architecture / benchmarks |
| Benefits | Emphasise product capabilities with specificity. "Processes 10K records in 4 seconds" > "fast and reliable" |
| CTA | Waitlist / early access framing. "Be the first to try it" / "Join the founding group" |
| FAQ | Focus on "why build this" and "how does it work" — vision-oriented questions |

### Early (Few Users)

Some signal, not yet proven at scale:

| Section | Adaptation |
|---------|-----------|
| Social Proof #1 | Beta user count, waitlist size. "340 teams signed up in the first week" |
| Social Proof #2 | 1-2 real testimonials from early users. Specific results even if small scale. |
| CTA | Early access / founding member framing. "Join as a founding member" |

### Growing (Some Traction)

Real proof available:

| Section | Adaptation |
|---------|-----------|
| Social Proof #1 | Customer logos (even 3-4), user count |
| Social Proof #2 | Multiple testimonials, one case study |
| CTA | Standard conversion CTA. "Start your free trial" |

### Established (Proven)

Full proof arsenal:

| Section | Adaptation |
|---------|-----------|
| Social Proof #1 | Logo bar, "As Seen In", specific stats |
| Social Proof #2 | Rich case studies, video testimonials, expert endorsements |
| CTA | Confidence-heavy. Stats near button, strong guarantee |

## Framework Application to Page Flow

### AIDA Flow (Cold Traffic, B2C)

```
HERO:     Attention — bold claim or question that stops the scroll
PROOF 1:  Interest — credibility that makes them want to learn more
BENEFITS: Desire — paint the "after" state they want to live in
PROOF 2:  Desire deepened — others already living it
FAQ:      Resolve remaining doubt
CTA:      Action — make it easy, make it now
```

### PAS Flow (Pain-Aware, B2B)

```
HERO:     Problem — name their specific pain
PROOF 1:  Agitate — "you're not alone, and it's costing you"
BENEFITS: Solution — how the product eliminates the pain
PROOF 2:  Solution validated — others solved it this way
FAQ:      Handle implementation/risk objections
CTA:      Relief — "stop dealing with [pain] today"
```

### Pierri Flow (B2B SaaS Positioning)

```
HERO:     [Segment] → [Context] → [Problem] → [Category] → [Capability]
PROOF 1:  Category validation — who else is in this space
BENEFITS: Feature → Capability → Benefit chains
PROOF 2:  Customer results within the category
FAQ:      Comparison, integration, migration questions
CTA:      Category-appropriate action
```
